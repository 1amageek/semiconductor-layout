## 依存関係と推奨順序

```
            M7 スケール・パリティ
           （可変空間索引の共通化）
            │           │
            │           └──────────────┐
            ▼                          ▼
  M8 ライブ LVS                M9 インタラクティブ配線
 （M3 接続性の上に             （M1 DRC / M2 DRD の上に
   デバイス抽出+照合）           経路探索 + shove）
            │                          │
            └────────┬─────────────────┘
                     ▼
            M10 階層編集
           （flatten ファンアウトが M7 の索引性能と
             M8/M9 のライブ判定に乗る）
```

- **M7 を最初に**: M8/M9/M10 はすべて「編集 tick ごとにライブ系が応答する」前提で設計されており、その応答時間は空間索引の規模耐性で決まる。M7 は M6 で実証済みのパターン（可変一様グリッド + 増分 apply）を検証側へ展開する基盤工事。
- M8 と M9 は M7 完了後なら並行開発可能（モジュール接点が分離している）。
- M10 は flatten が全ライブ系への唯一の入口（ファンネル）であることを利用するため、規模耐性（M7）が先にあると安全。

## 横断原則（全マイルストーン共通・交渉不可）

M1–M6 で確立した規律をそのまま適用する。

1. **単一編集ストリーム**: 人間のジェスチャも AI コマンドも新機能も、すべて `LayoutEditDelta` → `commitDelta` を通る。ライブ系への横流し・別経路を作らない。
2. **transient 経路は 2 本とも**: `commitDelta` と一時ジェスチャ状態（ドラッグ tick / mirror offset 等）の両方に新セッションの `apply` を配線する（M5 の教訓）。
3. **live == batch オラクル**: すべてのライブ判定は「ゼロから再計算した結果」とのマルチセット比較テストを持つ。ID では比較しない（flatten は再構築ごとに UUID を新規採番する）。
4. **silent fallback 禁止**: 予算超過・未対応ケース・外部ツール不在は、結果の中で正直に申告する（M6 の `stats.usedCellAggregates` 方式）。
5. **ベンチマーク規律**: `.serialized` + `.timeLimit`、`[bench]` 行で MEETS/MISSES を毎回印字、`#expect` キャップは観測値に余裕を持たせて回帰だけを落とす。
6. **最適化は計測から**: 推測で書き換えない。`sample <pid>` でホットパスを特定してから直す（M6 の CoW 教訓: 774s→1.5s の原因は最初の推測と違った）。
7. **struct ホットパスで self 変異クロージャ禁止**: `self` のメソッドが `self` を借用している間にクロージャで `self` を変異させると辞書全体の防御コピーが毎呼び出し発生する（M6 で実測 521 倍の差）。
8. **エラーは型付きで throw**: `try?` 禁止。復旧不能な編集は失敗として返し、ベースラインへ戻す（M2 の baseline-identity パターン）。
